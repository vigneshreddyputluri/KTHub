# KTHub Release 0.2 Delivery Plan

## Release Name

Clickable MVP

## Delivery Objective

Release 0.2 converts KTHub from a static concept prototype into a clickable MVP that demonstrates the core product workflows using mock data and a Node.js/Express backend foundation.

The release must preserve the existing visual identity, typography, spacing, and UI tone from Release 0.1 while improving navigation, task completion, and SDLC traceability.

## Business Goals

- Validate that users understand KTHub as a collaboration platform for knowledge transfer modules.
- Demonstrate learner and mentor KT view modes.
- Document user and institution account types with verified institution association.
- Show the main product workflows before database and authentication implementation.
- Establish backend project structure for Release 0.3.
- Maintain professional documentation for client review, delivery tracking, and future handover.

## Scope

### In Scope

- Client-side routing for primary product areas.
- KT detail screen for individual knowledge transfer modules.
- Create KT draft flow.
- Contribution proposal flow.
- Assignment curation flow.
- Mock learner/mentor view switching.
- Mock institution association request flow.
- Local mock persistence through browser localStorage.
- Express backend scaffold with health check and mock API endpoints.
- Updated SDLC documentation and release notes.

### Out Of Scope

- Real database persistence.
- Production authentication.
- Authorization middleware.
- File upload and media storage.
- Email notifications.
- LMS, SSO, or third-party integrations.
- Deployment pipeline.

## Account Types And View Modes Covered

- User account: represents a person, such as a student, lecturer, professor, alumni member, mentor, or professional.
- Institution account: represents a verified college, university, department, company, training center, or organization page.
- Learner view: explore KTs, view learning paths, and propose learner-centered improvements.
- Mentor view: review mentor notes, curate assignments, and propose guidance-centered improvements.
- Institution admin permission: detected from a verified institution membership held by a user account.

## Functional Requirements

| ID | Requirement | Release 0.2 Treatment |
| --- | --- | --- |
| FR-001 | Browse featured and recent KTs | Implemented in Explore route |
| FR-002 | Filter KTs by view mode, conceptual/technical category, difficulty, source | Implemented with mock data filters |
| FR-003 | View KT detail pages | Implemented as hash route detail view |
| FR-004 | Switch learner and mentor views | Implemented through view mode switcher |
| FR-005 | Publish a KT with structured metadata | Implemented as draft creation flow |
| FR-006 | Propose KT improvements | Implemented as contribution proposal flow |
| FR-007 | Review contributions | Represented as contribution activity state |
| FR-008 | Include practical assignments and rubrics | Implemented in detail and assignment curation flows |
| FR-009 | Institution collections | Represented through generic institution page and association model |
| FR-010 | Show trust signals | Implemented through rating, contributors, source, and version labels |
| FR-011 | Request institution association | Implemented as mock association request flow |

## Non-Functional Requirements

- Preserve Release 0.1 visual style and font stack.
- Keep the app runnable locally with simple commands.
- Maintain semantic documentation and release traceability.
- Keep frontend and backend loosely coupled through mock API shapes.
- Keep accessibility basics: labels, keyboard-friendly cards, semantic headings, and live updates.

## Acceptance Criteria

- User can navigate between Home, Explore, KT Detail, Create KT, Contributions, Assignments, and Institutions without page reloads.
- User can switch between learner and mentor view modes.
- User can open a KT detail page from Explore.
- User can create a KT draft and see it appear in the Explore list.
- User can submit a contribution proposal and see it in the contribution activity list.
- User can curate an assignment draft and see it in the assignments list.
- User can open Institutions, submit a mock association request, and withdraw that local request.
- Institutions page defaults to generic information when the user has no verified institution association.
- Express server starts locally and serves health and mock KT endpoints.
- Documentation clearly explains the scope, architecture, backlog status, and release notes.

## Delivery Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Scope expands into full backend MVP | Delays v0.2 | Keep database/auth out of scope until v0.3 |
| Static app becomes hard to maintain | Medium | Use modular JavaScript data/state/render functions |
| Mock flows are confused with production features | Medium | Label local/demo behavior in docs and code structure |
| UI drift from v0.1 | Medium | Reuse existing CSS variables, layout rhythm, and typography |

## Definition Of Done

- All v0.2 in-scope flows are clickable.
- The app can still be opened directly through `app/index.html`.
- Backend can be started with `npm start`.
- Docs are updated before release handoff.
- Git status contains only intentional v0.2 changes.
